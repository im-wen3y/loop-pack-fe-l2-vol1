## 레이어 이름 결정 — `_api` vs `_repository`

> 데이터 접근 폴더를 처음엔 `_repository`로 만들었다가 `_api`로 바꾼 이유.

### 고민 과정

1. `_repository/`에 `httpClient`(axios 인스턴스)를 두면서 "이건 `_utils`로 빼야 하나?" 고민 → `_utils`는 순수 함수 자리인데 httpClient는 설정을 든 인프라라 성격이 다르다고 판단, 그대로 데이터 계층에 둠.
2. 그 과정에서 "그럼 `_api`와 `_repository`는 뭐가 다르지?"라는 의문이 생김 → 두 계층의 정의를 비교(아래 표).
3. 정의에 비춰 `productRepository`를 다시 보니 URL을 직접 알고 DTO를 변환 없이 반환하고 있었다. **이름만 repository이고 하는 일은 api**였다.
4. "있어 보이는 이름"보다 "실제 동작"에 맞추기로 결정 → `_repository` → `_api`, `productRepository.ts` → `productApi.ts`로 변경.
5. 같은 맥락에서, 루트에 혼자 떠 있던 `queryClient`도 같은 성격(설정된 클라이언트)이라 `_api`로 함께 모음.

### 두 계층의 차이

| 구분    | api 계층              | repository 계층                    |
| ------- | --------------------- | ---------------------------------- |
| 관점    | 엔드포인트 호출(전송) | 데이터 출처를 숨긴 접근(도메인)    |
| 아는 것 | URL·메서드·DTO 모양   | "데이터를 가져온다"는 의도만       |
| 반환    | 서버 DTO 그대로       | 도메인 모델 (DTO→도메인 변환)      |
| 출처    | HTTP 고정             | 교체 가능 (HTTP·localStorage·캐시) |

- `api.ts`의 `getProductList`는 `/api/products` URL을 직접 알고 `GetProductListResponse`(DTO)를 변환 없이 반환한다 → 하는 일이 api 계층이다.
- repository라는 이름값을 하려면 DTO→도메인 변환이나 출처 추상화(예: wishlist를 localStorage·서버 공통 인터페이스로 감싸기)가 있어야 하는데, 지금은 출처가 HTTP 하나뿐이라 그 계층이 아직 필요 없다.
- 이름을 실제보다 있어 보이게 붙이면 나중에 혼란을 주므로 동작에 맞춰 `_api`로 통일. 도메인 변환·다중 출처가 생기는 시점에 `_repository`를 별도로 도입 검토.
- 쿼리 클라이언트(`queryClient`)는 이 계층이 쓰는 "설정된 인스턴스"라 `_api`에 함께 둠 — 순수 함수가 아니므로 `_utils` 대상은 아님. (전송 클라이언트 axios `httpClient`는 처음엔 함께 뒀으나 mock 호환 문제로 제거 — 아래 근거 섹션.)

## HTTP 클라이언트 — 왜 axios가 아니라 native `fetch`인가

> 처음엔 `_apis/httpClient.ts`에 axios 인스턴스(`adapter: 'fetch'`)를 두고 `getProductList`가 그걸 쓰게 했다. 그런데 상품 목록이 로드되지 않아 디버깅한 끝에 axios를 걷어내고 native `fetch`로 되돌렸다. (그 결과 `httpClient.ts`는 삭제)

- mock(`installMockApi`, 스타터 파일이라 수정 불가)은 `window.fetch`를 가로채되 `url.startsWith('/api/products')`처럼 **상대경로 prefix**로만 매칭한다.
- axios의 fetch 어댑터는 요청을 `Request` 객체로 감싸는데, `Request` 생성자가 상대경로를 `http://localhost:5173/api/products?...` 같은 **절대 URL**로 정규화한다. → mock의 `input.url`이 절대경로가 되어 `startsWith('/api/products')`가 false → mock 분기를 못 타고 실제 네트워크로 새어나가 목록이 안 떴다. (`installMockApi()` 자체는 정상 실행돼 있었다 — "호출은 되는데 조건에서 걸러지는" 문제였다.)
- 해결: `getProductList`에서 axios 대신 상대경로 문자열을 그대로 넘기는 `fetch`를 쓴다. 문자열이면 `Request`로 감싸이지 않아 상대경로가 유지돼 mock이 정상 가로챈다. `response.ok`로 에러를 명시 처리하고 `response.json()`으로 DTO를 반환한다.
- 계층 책임은 그대로다 — axios가 하던 `.data` unwrap을 이제 `getProductList`가 `response.json()`으로 직접 하므로, 호출부(`_service`)는 여전히 전송 세부(`Response` 객체)를 모른다.
- 트레이드오프: axios의 인터셉터·자동 JSON 파싱·에러 정규화는 포기했다. 지금은 요청이 GET 하나뿐이라 native `fetch`로 충분하고 mock 환경 호환이 우선이었다. 실제 백엔드로 붙어 공통 처리(인증 헤더·에러 변환)가 필요해지면 그때 클라이언트 재도입을 검토한다.

## 레이어 경계 — `_apis` vs `_service`

> `_api`(단수, `productApi.ts` 한 파일)에서 도메인별 하위 폴더 `_apis/product/`(`api.ts` + `model.ts`)로 나뉘고, TanStack Query를 다루는 `_service/product/`(`queries.ts` + `service.ts`)가 그 위에 새로 얹혔다. 두 계층이 서로 무엇을 알고 모르는지 정리한다.

| 구분          | `_apis` 계층                                                                          | `_service` 계층                                                                           |
| ------------- | ------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| 관점          | 엔드포인트 호출 — URL·메서드·쿼리스트링 조립                                          | 그 호출을 언제·어떻게 캐싱할지                                                            |
| 아는 것       | URL, HTTP 메서드, 요청/응답 DTO(`model.ts`)                                           | queryKey 구조, 캐싱 전략(`placeholderData` 등), 어떤 api 함수를 호출할지                  |
| 모르는 것     | TanStack Query의 존재 — `getProductList`는 Query 없이도 재사용 가능한 순수 async 함수 | 실제 HTTP 요청 방법 — `queries.ts`는 `fetch`를 직접 부르지 않고 `getProductList`만 import |
| 반환          | DTO(`GetProductListResponse`) 그대로                                                  | TanStack Query state(`data`/`isLoading`/`error` 등)                                       |
| 컴포넌트 접점 | 직접 참조하지 않음 — `_service`를 거쳐서만 쓰인다                                     | `useGetProductList` 훅 하나만 노출                                                        |

- `api.ts`(`getProductList`)는 `fetch` 응답을 `response.json()`으로 풀어 DTO로 반환한다 — 호출부가 `Response` 객체를 몰라도 되게 하는 것도 이 계층 책임.
- `queries.ts`의 `productQuery.list()`는 `queryOptions()`로 만들어 `useQuery`/`useSuspenseQuery`/`prefetchQuery` 등 여러 훅에서 재사용 가능한 "캐싱 전략 선언"만 갖는다. 실제 훅 호출(`useQuery(...)`)은 `service.ts`가 맡는다.
- 도메인이 늘어나면 `_apis/<domain>/`, `_service/<domain>/`이 나란히 늘어나는 구조 — `_api`(단수)가 이미 담고 있던 "여러 도메인의 api 함수를 한 파일에 몰아넣는" 문제를 폴더 단위로 미리 막는다.
- `ProductListPage.tsx`는 `useGetProductList`로 전환 완료 — 다만 컴포넌트가 직접 `useGetProductList`를 호출하지 않고 `_hooks/productList/useProductListQuery`를 거친다. Service(`useGetProductList`)는 "캐싱"만, 파라미터 조립·페이지 계산 같은 화면 관심사는 `_hooks` 계층이 맡아 경계를 한 겹 더 나눴다.

## component-review 반영 — 4개 관점 리뷰 이후 구조 변경

> `/component-review`로 `src/productList/`, `src/_hooks/productList/`, `src/_apis/`, `src/_service/`를 리뷰해 10건을 발견했고, 그중 심각도 상위 8건을 반영했다. 나머지 2건(`parsePrice` 분리, `response.json()` 런타임 검증)은 위 표의 근거를 그대로 유지하거나 새 의존성 추가라 이번 단계 제외로 결정했다.

### SearchFilter props 그룹화 — 훅 추출만으로 부족했던 이유

리뷰 이전까지 이 문서(위 13행)는 "`useProductFilters` 훅 추출 시 props 9개가 줄어들 예정"이라고 적었지만, 실제로는 줄지 않았다. 훅 추출은 **상태의 소유권**을 페이지에서 훅으로 옮겼을 뿐, `SearchFilter`가 받는 **prop의 개수**는 그대로였다 — 페이지가 훅에서 값 4개·핸들러 5개를 꺼내 개별 prop으로 다시 펼쳐 넘겼기 때문이다. "책임을 훅으로 옮기면 컴포넌트도 저절로 얇아진다"는 예상이 틀렸던 지점.

해결: `SearchFilter`의 props를 `filters`(값 4개를 담은 객체) / `onFilterChange`(핸들러 4개를 담은 객체) / `onResetFilters` 3개로 그룹화했다. 각 필터가 독립적으로 바뀌므로 Context 도입은 과함(1단계 props drilling이라 `component-design.md`의 "3단계 이상" 기준 미달) — 객체 그룹화로 `component-design.md`의 "props 5개 초과 시 재검토" 기준을 충족했다.

### useProductListQuery — 파라미터 조립을 순수 함수로 분리

기존에는 `useProductListQuery`가 (1) `ProductFilters` → `GetProductListParams` 변환 (2) `useGetProductList` 호출 (3) `totalPages` 파생 계산을 한 훅에서 수행해, "그리고"가 두 번 들어가는 설명이 됐다(component-review의 Custom Hook 책임 관점 기준 위반).

파라미터 변환 부분(순수 계산)을 같은 파일 안의 모듈 레벨 `function toGetProductListParams(filters, page)`로 분리했다. `code-style.md`의 "function 선언은 컴포넌트/훅 외부 유틸 함수에만" 규칙에 부합하는 지점이라 `_utils`로 파일을 옮기지 않고도(왕복 indirection 없이) 훅 본문을 얇게 만들 수 있었다. 훅은 이제 "변환된 파라미터로 조회하고 파생값을 반환한다"는 한 문장으로 설명된다.

같은 김에 `refetch`도 반환값에 추가했다 — `ProductListPage.tsx`의 에러 재시도가 `window.location.reload()`(TanStack Query 캐시를 포함해 앱 전체를 리로드) 대신 `refetch()`(캐시 유지한 채 해당 쿼리만 재조회)를 쓰도록 바꾸는 데 필요했다.

### usePagination — 스크롤 부수효과를 제네릭 훅으로 분리

`usePagination`이 페이지 상태 관리와 "페이지 변경 시 스크롤 맨 위로" 부수효과라는 별개 관심사를 한 훅에 담고 있어, 스크롤이 필요 없는 곳(예: 모달 내 페이지네이션)에서 재사용하기 어려웠다.

`useLocalStorageState` → `useWishlist`/`useRecentlyViewed`에서 이미 쓴 "제네릭 훅 + 도메인 훅 조합" 패턴을 그대로 따라, `_hooks/useScrollToTop.ts`(도메인 무관, dependency 하나만 받는 제네릭 훅)로 분리했다. `usePagination`은 이제 `page`·`goToPage`·`resetPage`만 남았고, `ProductListPage.tsx`가 `useScrollToTop(pagination.page)`를 별도로 호출해 조합한다.

### useProductFilters — 핸들러 중복 제거

6개의 개별 `useState`(category/minPrice/maxPrice/sortBy/searchQuery/inStockOnly)와, 그만큼 반복되던 `setX(value); onFilterChange()` 패턴을 하나의 `ProductFilters` 객체 상태 + 제네릭 `updateFilter(key, value)` 헬퍼로 통합했다. 각 `handleXChange`는 `updateFilter('X', value)` 한 줄로 줄었다. 훅의 반환 형태(개별 핸들러 8개)는 호출부(`ProductListPage.tsx`, `SearchSort.tsx`) 호환을 위해 그대로 유지했다 — 내부 구현만 바뀌었을 뿐 외부 계약은 깨지 않았다.

### useProductListQuery — 검색 debounce 추가 (self-review 버그 헌팅 반영)

셀프 리뷰에서 검색어 입력마다 debounce 없이 즉시 재조회되는 걸 지적받아, `useDebouncedValue(filters.searchQuery, 300)`로 조회용 검색어만 debounce했다. 입력창(`filters.searchQuery`)은 그대로 즉시 반영해 타이핑 반응성은 유지하고, 네트워크 요청만 300ms 정지 후 1회로 줄였다 — Playwright로 타이핑 중 0건·정지 후 1건을 확인.

### 검증

`tsc --noEmit`·`pnpm lint` 통과 후, Playwright로 dev 서버를 직접 구동해 브라우저에서 확인했다: 필터 그룹화 후에도 카테고리·가격·재고 필터 정상 동작, 페이지 이동 시 스크롤 맨 위 복귀(`usePagination`/`useScrollToTop` 분리 후에도 유지), 에러 재시도가 `window` 마커 생존 여부로 하드 리로드 없이 `refetch`만 발생함을 확인. 검색 debounce는 타이핑 중 API 호출 0건, 입력 정지 300ms 후 1건 발생을 확인.

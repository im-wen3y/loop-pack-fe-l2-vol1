# 컴포넌트 리뷰 체크리스트 & 출력 형식

## Task 1 — 관심사 분리

- JSX와 API 호출이 한 파일에 섞여 있는가
- 비즈니스 로직(필터/검색/계산 등)이 Custom Hook으로 분리되었는가
- 순수 함수(포맷/계산 등)가 `utils`로 분리되었는가
- 컴포넌트가 UI 렌더링에만 집중하는가
- 파생값을 `useEffect`로 동기화하지 않고 `const`로 계산하는가

## Task 2 — Custom Hook 책임

- 한 문장으로 설명되는가 ("그리고"가 두 번 들어가면 분리 후보)
- 같은 상태+로직 조합이 반복되는가 (추출 후보)
- JSX나 스타일을 반환하고 있지 않은가
- `fetch`/`axios` 구현체에 직접 묶이지 않고 service 함수에 의존하는가
- Hook이 컴포넌트의 복잡도를 그대로 옮긴 "큰 Hook"은 아닌가
- Hook 이름이 역할을 충분히 설명하는가 (`useData` ❌ / `useProductFilter` ✅)

## Task 3 — API 레이어

- endpoint와 request/response 변환이 한 곳에 모여 있는가
- 컴포넌트가 서버 응답의 세부 구조를 과하게 알고 있지 않은가
- 서버 상태가 일관된 패턴으로 다뤄지고 있는가

## Task 4 — 분리 근거 / 프로세스

- "이 분리가 무엇을 좋게 하나"가 문서(README 등)나 커밋/PR 설명에 기록되어 있는가
- 분리하지 않은 부분에도 근거가 있는가 (분리 자체가 목적이 되는 과잉 분리 방지)
- 타입·린트를 `any`/`@ts-ignore`/`eslint-disable`로 침묵시키지 않았는가

## 출력 형식

발견 사항을 아래 우선순위로 분류하여 출력한다. 해당 없는 등급은 생략.

```
### 🔴 Critical
동작이 깨지거나 데이터 손실 가능성이 있는 문제

### 🟠 Major
관심사 분리·설계에 영향을 주는 문제

### 🟡 Minor
가독성·명명 등 사소한 문제

### 💡 Suggestions
필수는 아니지만 개선하면 좋은 제안
```

각 항목은 `파일:라인 — 설명 (해당 관점)` 형식으로 작성한다.

리포트 마지막에 4관점 통과 요약표를 붙인다:

```
| 관점 | 결과 |
|------|------|
| 관심사 분리 | ✅ / ❌ |
| Custom Hook | ✅ / ❌ |
| API 레이어 | ✅ / ❌ |
| 분리 근거/프로세스 | ✅ / ❌ |
```

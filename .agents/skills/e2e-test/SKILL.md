# E2E 테스트

사용자에게 보이는 핵심 흐름과 실제 브라우저 경계에서 발생할 수 있는 실패를 함께 검증한다. 해피 케이스만 작성한 상태를 완료로 판단하지 않는다.

## 작성 절차

1. 관련 화면, 상태, API, 기존 E2E 테스트를 먼저 확인한다.
2. 핵심 사용자 흐름의 정상 시나리오를 작성한다.
3. 각 기능에서 발생 가능한 비정상 시나리오를 다음 기준으로 점검한다.
   - 잘못되거나 비어 있는 입력
   - 최소·최대·범위 밖의 값
   - API 실패, 지연, 재시도와 복구
   - 새로고침, 뒤로·앞으로 이동, 직접 진입
   - debounce·비동기 요청 중 조건 변경이나 페이지 이탈
   - localStorage 등 영속 상태의 손상과 버전 변경
   - 결과 없음, 중복 조작, 빠른 연속 조작
4. 기능별로 관련 있는 오류·경계·복구 시나리오를 최소 1개 이상 작성한다.
5. 오류를 표시하는 데 그치지 말고 가능한 경우 재시도나 정상 상태 복귀까지 검증한다.
6. Chromium과 WebKit에서 독립 실행될 수 있도록 테스트 사이의 상태 의존성을 제거한다.

## 필수 규칙

- 정상 시나리오만 추가하고 작업을 완료하지 않는다.
- 구현 세부 사항 대신 사용자가 보는 결과, URL, 접근성 상태와 상호작용을 검증한다.
- locator는 `getByRole`과 접근 가능한 이름을 우선 사용한다.
- 고정 지연은 debounce 취소처럼 시간 경계 자체가 검증 대상일 때만 최소한으로 사용한다.
- API 오류·지연은 Playwright route interception으로 재현 가능하게 만든다.
- 테스트할 수 없는 오류 시나리오가 있다면 생략하지 말고 이유, 남은 위험, 대체 검증을 완료 보고에 남긴다.

## 완료 점검

- 핵심 해피 케이스가 있는가?
- 입력과 URL의 경계값을 검증했는가?
- 네트워크 실패 또는 지연을 검증했는가?
- 새로고침·탐색·영속 상태 중 관련 있는 항목을 검증했는가?
- 오류 후 복구 경로를 검증했는가?
- 각 테스트가 다른 테스트에 의존하지 않는가?

하나라도 관련 있는데 빠졌다면 테스트를 보강한 뒤 완료한다.

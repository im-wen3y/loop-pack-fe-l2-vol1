# 셀프 리뷰

TRIGGER — 셀프 리뷰, self-review, PR 전 점검 요청 시 이 스킬을 먼저 로드.

## 5단계 체크

1. **동작** — happy path뿐 아니라 로딩/에러/빈 상태 + 상태 전이(적용→해제, 선택 변경)까지 머릿속에서 돌려봤나
2. **의존성** — import한 패키지가 실제 존재하나? 기존 유틸·타입 두고 중복 생성 안 했나? hook 의존성 배열 완전한가?
3. **타입·린트 침묵** — 새 `any`/`as`/`@ts-ignore`/`eslint-disable`이 없나 (있다면 전부 "기계가 경고하려던 자리")
4. **표면적** — `dangerouslySetInnerHTML`, 입력 무검증 렌더링 같은 XSS 구멍 없나
5. **Next 경계** — `'use client'`가 실제 필요(훅/이벤트 핸들러/브라우저 API)와 어긋나지 않는가(불필요한 선언 또는 누락), 데이터 조회 훅이 service 함수를 거치는가(Server Component 직접 fetch는 예외), mutation을 Server Actions 대신 불필요하게 클라이언트에서 처리하고 있지 않은가

## 실행 방법

1. `git diff main...HEAD`로 전체 변경 사항 수집
2. 해당 주차의 `docs/week-XX/checklist.md`가 있으면 함께 읽어 체크리스트 항목 충족 여부도 리뷰에 포함한다
3. 5단계를 각각 독립적으로 리뷰한다. 실행 환경이 병렬 에이전트를 지원하면 단계별로 병렬 점검한다
   - 동작 점검
   - 의존성 점검
   - 타입·린트 침묵 점검
   - 표면적 점검
   - Next 경계 점검
4. 각 단계의 결과를 종합하여 우선순위별 리포트 생성

## 출력 형식

발견 사항을 아래 우선순위로 분류하여 출력한다. 해당 없는 등급은 생략.

```
### 🔴 Critical
동작이 깨지거나 데이터 손실 가능성이 있는 문제

### 🟠 Major
버그는 아니지만 설계·안전성에 영향을 주는 문제

### 🟡 Minor
코드 품질·가독성에 영향을 주는 사소한 문제

### 💡 Suggestions
필수는 아니지만 개선하면 좋은 제안
```

각 항목은 `파일:라인 — 설명 (해당 단계)` 형식으로 작성한다.

리포트 마지막에 5단계 통과 요약표를 붙인다:

```
| 단계 | 결과 |
|------|------|
| 동작 | ✅ / ❌ |
| 의존성 | ✅ / ❌ |
| 타입·린트 침묵 | ✅ / ❌ |
| 표면적 | ✅ / ❌ |
| Next 경계 | ✅ / ❌ |
```
